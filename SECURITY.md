# Security Policy

## Supported versions

Security fixes are considered for the latest released version and the default branch. Older benchmark releases are immutable research artifacts and may not receive backports.

## Reporting a vulnerability

Please use [GitHub private vulnerability reporting](https://github.com/EvanGribar/SpecBench/security/advisories/new) when available. Do not open a public issue for a suspected vulnerability.

Include the affected version or commit, affected file or command, reproduction steps or proof of concept, impact, and any safe mitigation. Do not include API keys, access tokens, personal data, or provider request headers.

Response time and remediation availability depend on maintainer capacity; this project does not promise a fixed SLA. We will acknowledge reports when practical and keep follow-up private until disclosure is safe.

If an API key or token is accidentally exposed, revoke it immediately with the provider, then report the location privately. Do not wait for repository cleanup.

Disagreements about benchmark cases, scoring, model behavior, or published results are research and methodology discussions, not security vulnerabilities. Use the relevant issue template instead.
