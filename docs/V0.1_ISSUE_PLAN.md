# SpecBench v0.1 Issue Plan

## Definition of done

v0.1 is complete when a clean Node 20+ checkout can install with pnpm, validate
eight benchmark cases, run fixture reviewers, score their outputs, and generate
JSON, terminal, and standalone HTML reports; the documented tests pass without
network access or provider credentials.

## Seed-case acceptance criteria

Each of the first three cases must use a different primary category and include:

- A human-readable product requirement with an identifier.
- A baseline commit/revision, review patch, and valid changed-file locations.
- At least one intentional violation with stable finding id, severity, explanation,
  requirement reference, and acceptable aliases.
- A baseline application test that demonstrates the intended product rule, or an
  explicit reason a test cannot capture the behavior.
- Fixture outputs for perfect detection, false positives, missed findings,
  duplicates, total failure, and wrong severity across the suite.

Proposed first three categories: administrator-only invitations (authorization),
plan-specific seat limits (subscription), and idempotent notification cancellation
(state transition/cancellation). The exact patches are deferred until issues 2–8
have fixed their inputs and output formats.

## Consistency checks completed before case authoring

- The case schema in `SPEC.md` uses JSON, explicit schema versions, and the
  fields required by the matcher.
- The architecture gives `core` ownership of validation and the scorer ownership
  of matching/metrics, avoiding circular package dependencies.
- The CLI order matches the artifact flow: validate → run → score → report.
- Fixture-only CI and explicit live-call controls satisfy the local-first cost
  constraint.

## Non-goals for this milestone

No SaaS dashboard, authentication product, cloud database, submission portal,
paid infrastructure, or automatic model evaluation is included in v0.1.
