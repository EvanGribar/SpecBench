# SpecBench v0.3 experiment readiness

## Inspection

On 2026-07-18, the v0.2 suite was validated with `pnpm validate` (10 cases), `pnpm typecheck`, and `pnpm test`. Each case was inspected for a requirement, seeded violation, impact statement, aliases, distractor, expected-finding identifier, changed-file reference, and valid line range. The scorer was inspected for one-to-one finding matching and false-positive treatment. Adapters and run/report schemas were inspected for runtime and cost propagation; CI runs only typecheck, validation, and fixtures.

## Defects found and corrected

- The case files referenced `reference-saas-v0.2`, which was not initially present as a local Git ref. An annotated immutable tag now resolves it to the v0.2 reference baseline commit, and validation fails when a case baseline cannot be resolved.
- The v0.2 runner did not preserve repetition identity, timestamps, failures, raw output, or partial runs as distinct artifacts. The v0.3 runner assigns UUID-backed run IDs, writes one file per repetition, retains failure records, and refuses to overwrite outputs.
- The previous configuration files did not enforce controlled parity. The three v0.3 configurations now share model, token cap, case ids, output contract, prompt version, and threshold; debate rounds are the intended swarm comparison variable.

## Remaining methodological risks

The 10 cases come from one reference SaaS and the same project team built the cases and matcher. Alias and file/line matching may still misclassify valid findings, so human adjudication remains required before strong claims. Provider pricing is not inferred when unavailable; results retain measured usage and may have an unknown cost. Three repetitions are a small sample, and the controlled swarm prompt is an experimental implementation rather than a claim about every multi-agent system.

## Readiness decision

**Ready for a controlled preliminary run.** Run `pnpm experiment:v0.3:dry-run` first, then explicitly provide `OPENAI_API_KEY` for live execution. A complete release still requires saved runs for all configurations, scoring, adjudication, reports, and the release-criteria audit.
